What is it
As part of our migration from CloudBees (Jenkins) to GitHub Actions, we propose establishing a centrally managed GitHub Actions Runner platform within a dedicated AWS account. This becomes the standard execution layer for all CI/CD workloads, ensuring a consistent, secure, and scalable foundation as we transition away from legacy Jenkins infrastructure.

How we do it
The Platform Engineering team will own and operate the GitHub Actions runner platform (using Kubernetes/EKS-based autoscaling runners), including infrastructure, security baselines, and access controls. Engineering teams will consume runners directly via GitHub workflows without managing any underlying compute. Existing ad-hoc or team-specific runner setups will be consolidated into this model to ensure uniform governance, isolation, and scalability across teams and clients.

Advantages (and why not alternatives)
This approach provides a controlled, enterprise-grade execution environment aligned with GitHub Actions, improving security (ephemeral runners, IAM-based access), reducing cost through shared autoscaling infrastructure, and accelerating delivery by removing operational overhead from teams. Continuing with existing or decentralized runner implementations would perpetuate inconsistency, duplicated cost, and security gaps, and would not meet the standardisation and compliance requirements expected from the Jenkins-to-GitHub migration.

Common SRE Approach — First Call Summary
1. Objective
Establish a common reliability operating model across multiple teams and platforms
Ensure consistency, visibility, and supportability without disrupting existing ways of working
2. Key Principles
Not a one-size-fits-all model
Define common minimum standards, not identical implementations
Respect platform diversity (AWS, Azure, Snowflake, Fivetran, etc.)
Focus on outcomes (reliability, operability), not tools or process overhead
3. Problem We Are Solving
Fragmented operational practices across teams
Inconsistent incident management and escalation
Limited visibility into service health and dependencies
Reactive firefighting vs proactive reliability engineering
No unified reliability metrics or reporting
4. What “Common SRE” Means
Common across all teams:
Service ownership model
Incident severity & escalation standards
Minimum monitoring & alerting baseline
SLI/SLO framework
Post-incident review practices
Flexible per team:
Tooling and implementation choices
Platform-specific observability patterns
Deployment and support models
5. Phased Approach
Stage 1 — Discover & Baseline
Assess current maturity, tooling, and processes
Identify risks, gaps, and quick wins
Stage 2 — Define Standards
Establish common SRE framework and minimum standards
Define SLOs, incident model, ownership
Stage 3 — Uplift in Waves
Improve teams based on criticality and risk
Enhance observability, alerts, runbooks, automation
Stage 4 — Optimize & Govern
Introduce proactive reliability practices
Continuous improvement and executive reporting
